# Apps

Installable business-vertical Apps. Phase 3 adds the first one, `restaurant` (see `docs/phases/PHASE_3_PLAN.md`) -- a counter-service order-taking App built entirely on Core's existing Order and Inventory Engines, proving the full Platform chain (App Registry → Platform entitlement → Core → Audit/Notifications) end-to-end. Phase 4.1 adds the second, `retail` (see `docs/phases/PHASE_4_PLAN.md`) -- proving the same architecture needs zero further Core changes for a second vertical. Retail is the minimal case: since payment/tender is out of scope, Retail has no data Core doesn't already model, so it has no `application/*.repository.ts` and no App-owned Firestore collection at all -- its `application/sale.service.ts` calls Core's Order Engine directly. Phase 4.2 adds `loyalty` (see `docs/phases/PHASE_4_2_PLAN.md`) -- a cross-cutting App (not tied to one vertical) that reacts to completed orders from *either* Restaurant or Retail by reading Core's own audit log, never importing either App or Core's order-engine's internals directly, proving Apps can observe each other's activity only through Core's own durable, already-existing record of it. Phase 6 (see `docs/phases/PHASE_6_PLAN.md`) adds three more: `barcode` (the same minimal shape as Retail -- no collection, scan-to-lookup/scan-to-sell calling the Inventory/Order Engines directly), `kitchen-display` (a realtime board -- its own Client Component reads Firestore directly via the client SDK, the first App to do so, bridged to a real Firebase Auth identity by `core/auth`'s `mintClientAuthToken`, gated entirely by the existing `firestore.rules`), and `ai-assistant` (a read-only Q&A layer grounded only in the asking user's own already-capability-gated Core reads, with the one new external dependency an LLM call inherently requires).

Each App conforms to `src/app-registry`'s `AppManifest` type (`id`, `displayName`, `icon?`, `routeKey`) and depends on Core's public interface and Shared only. `routeKey` is plain data -- the React component it maps to lives at the Next.js route layer (`src/app/(dashboard)/[companyId]/apps/[appId]/[[...slug]]/app-roots.ts`), never in App Registry itself.

Each App's own internal shape (mirrored by `restaurant`; `retail`/`barcode` omit the repository, having no collection of their own):
- `manifest.ts` -- registers with App Registry (imported once, by `app-registry/registry.ts` itself).
- `domain/` -- the App's own types, never duplicating a Core field.
- `application/` -- `*.repository.ts` (raw Firestore reads/writes to the App's own collections only, where an App has any) and `*.service.ts` (business rules, plain async functions, callable by a future API/CLI/job the same way Platform's services are).
- `actions.ts` -- Server Actions only: CSRF check, zod parse, call the App's own service, map errors, `revalidatePath`.
- `components/` and `routes/` -- UI, never the Admin SDK (enforced by `tests/architecture/apps-ui-no-firestore.test.ts`); call the App's own application-layer services, never Firestore's Admin SDK directly. Kitchen Display's `KitchenBoard.tsx` is a deliberate, narrow exception: a Client Component reading via the *client* Firestore SDK for a realtime feed, authorized entirely by `firestore.rules` rather than a Server Component/Action -- see `PHASE_6_PLAN.md` §3 for the full reasoning, including a build-time lesson (a Client Component must import a Server Action from its own source file, e.g. `@/core/auth/actions`, never through the general `@/core` barrel, or the entire server-only module graph gets bundled for the browser).

Import-boundary rules: Apps may depend only on Core's public interface, App Registry (own manifest), and Shared; Apps must not import Connectors or Platform directly; Apps must not import each other's internals.

An App's own Firestore-owned data lives under `companies/{companyId}/apps/{appId}/{collection}` -- nested under the same document Platform's install-state doc (`enabled`/`installedAt`/`config`) already owns for that App, since a Firestore document can carry both its own fields and its own subcollections.
