# Core

The permanent, business-agnostic platform.

**Implemented:**
- `auth` (Phase 1B) — Firebase Authentication (email/password), server-side session cookies, CSRF protection, basic rate limiting. Produces only a verified Firebase Auth UID + session; no Firestore data.
- `companies` (Phase 1C) — the Multi-Tenant Organization Model: the onboarding transaction (Company + default Branch + Owner Membership, atomic), `requireCompanyMembership()` as the single authorization entry point for anything company-scoped, and the membership-lookup helpers (`getMembership`, `listMyCompanies`, `hasBranchAccess`).
- `users` (Phase 1C) — the `users/{uid}` profile document (created only by onboarding's transaction) and a self-only `displayName` update.
- `roles-permissions` (Phase 1D) — the capability matrix (`ROLE_CAPABILITIES`) and the guard functions (`hasCapability`, `requireCapability`, `outranks`, `isSuperAdmin`) every module calls to answer "can this actor do this action." `companies/members-actions.ts` (role changes, deactivation) is the first consumer.
- `inventory-engine` (Phase 1E) — the one Core module with real `domain`/`application`/`infrastructure` layering (per `ARCHITECTURE.md` §4). Items are company-wide; stock is per-branch (`${branchId}_${itemId}` keyed) and gated by `hasBranchAccess()` on top of the `inventory.view`/`inventory.write` capabilities. Every stock mutation (`receiveStock`, `wasteStock`, `adjustStock`, `recordStockCount`, `transferStock`) runs inside a Firestore transaction and writes an immutable `inventoryMovements` entry alongside the stock update — no direct writes to `quantityOnHand` exist anywhere else. Plain server-only functions, not form-bound Server Actions — no UI consumes this yet (see `docs/phases/PHASE_1E_PLAN.md` §5). Also exposes `planStockChange`/`commitStockChangePlan` (1F), a two-phase split of the same transaction body so another engine can process *multiple* items' stock changes in one transaction without violating Firestore's read-before-write rule.
- `order-engine` (Phase 1F) — same layering as Inventory Engine. `pending → completed → voided` lifecycle (`canTransition()` is the single source of truth, and the idempotency guard: re-completing an already-completed order throws rather than double-deducting stock). `completeOrder()`/`voidOrder()` never write to `stock`/`inventoryMovements` directly — they plan and commit every line's stock change through Inventory Engine's `planStockChange`/`commitStockChangePlan`, in the same transaction as the order's own status update. `orders.void` is capability-gated separately from `orders.create`/`orders.complete` (Owner/Manager only vs. all four roles).
- `audit-logs` (Phase 1G) — `writeAuditInTransaction()`, the single write-through logging primitive every mutation path in 1C–1F calls, always inside the same Firestore transaction as the mutation it records (never a separate best-effort write). `listAuditLogs()` is gated by the new `audit.view` capability (Owner/Manager only). Closing the one gap this required: `companies`' rename/suspend was previously a direct, capability-gated client write (1D) with no server-side interception point to log from — 1G adds `updateCompanyAction`/`suspendCompanyAction` (`core/companies/company.ts`) to route both through Admin SDK + `writeAuditInTransaction`, and `firestore.rules` now denies all direct `companies` writes unconditionally. `AuditAction` is not a single hand-maintained union: it's assembled from each owning module's own action type (`CompanyAuditAction`/`MembershipAuditAction` in `core/companies/types.ts`, `InventoryAuditAction` in `core/inventory-engine/domain/types.ts`, `OrderAuditAction` in `core/order-engine/domain/types.ts`) — a new mutation's action literal is added right next to the enum/union that already drives it (e.g. `MovementType`), never by editing `core/audit-logs` itself. `listAuditLogsPage()` (added ahead of any real UI) is the cursor-paginated counterpart to `listAuditLogs()` — same signature-stable `{ items, nextCursor }` shape as `listNotificationsPage()` below, so the eventual audit-log screen needs no breaking API change.
- `notifications` (Phase 1G) — a thin channel abstraction (`createNotification`/`createNotificationInTransaction` dispatch to `channels/in-app.ts`, the only channel implemented so far) so a future email/SMS/WhatsApp channel is additive, not a rewrite. Per-user, not per-company (`users/{uid}/notifications`) — `listNotifications`/`markAsRead`/`markAllAsRead` operate on the caller's own uid, trusted the same way `core/users/profile.ts`'s `getUserProfile(uid)` is. `members-actions.ts`'s role-change/deactivation actions are the first callers, notifying the affected member in the same transaction as the membership update and its audit log entry. `listNotificationsPage()` is the cursor-paginated counterpart to `listNotifications()`, sharing `src/lib/firebase/pagination.ts`'s `applyCursor()`/`DEFAULT_PAGE_SIZE` with `core/audit-logs`. `markAllAsRead()` splits its writes into `<=500`-operation `WriteBatch`es (Firestore's hard cap on a single batch) rather than assuming a user's unread count ever stays under that limit.
- `companies/company-settings.ts` (Phase 2) — tenant configuration data (`companies/{companyId}/settings/{settingId}`, one document per category), kept in Core rather than Platform since it's business-agnostic company data, not a commercial concept. Phase 2 implements only `branding` (`logoUrl`/`primaryColor`); future categories (localization, tax, numbering, receipts, currencies, printing, regional) each get their own settingId document, never a merged one, and never a field on `Company` itself (which stays exactly as it was at the end of 1G).

**Additive, non-breaking Phase 2 changes to existing modules:**
- `roles-permissions/guard.ts` gains `requireSuperAdmin()` — the first Super Admin *write* path in the system (every prior use of `isSuperAdmin()` was a read-only rules bypass), reusing the existing predicate. Deliberately narrow: used by exactly one action (`platform/app-installs`' `forceToggleApp`), not a general bypass. `Capability`/`ROLE_CAPABILITIES` themselves are untouched — Platform's own capabilities never leak into Core's matrix (see below).
- `audit-logs/audit-logger.ts`'s `writeAuditInTransaction`/`AuditLogParams` are generalized to accept an optional `<TAction, TTargetType>` type parameter, defaulting to Core's own closed `AuditAction`/`AuditTargetType` unions. Every existing Core call site is unaffected (no type argument supplied); Platform calls it with its own closed types (e.g. `writeAuditInTransaction<AppInstallAuditAction, "app">(...)`), getting full compile-time exhaustiveness over its own vocabulary without `core/audit-logs` ever importing or knowing Platform's action literals exist. `AuditLogEntry.action`/`.targetType` (the read side) are widened from the closed unions to `string`, honestly reflecting that the log now legitimately contains entries from more than one layer.

**Reserved for a future need, not built now:** if a Core primitive ever needs to ask Platform something (e.g. "is this feature active for this company?"), the rule is that Core accepts the answer via an injected function parameter (a port), never by importing Platform directly. No Core function needs this in Phase 2, so no such port exists yet — documented here so it isn't improvised inconsistently later.

Import-boundary rule: Core must never import from `src/apps`, `src/connectors`, `src/platform`, or `src/app-registry`.

---

Phase 2 also activates four modules **outside** Core (see `docs/phases/PHASE_2_PLAN.md` for the full architecture, dependency rules, and rationale):
- **`src/platform`** — commercial/tenant-activation business logic only (license entitlement, installed-App state + rules, connected-Connector state + rules). Repositories and services, never a Server Action, form, or `"use server"` directive — verified by a permanent test (`tests/architecture/platform-no-server-actions.test.ts`). Depends on Core, App Registry, and Connectors; never on Apps or Settings.
- **`src/app-registry`** — a pure, zero-runtime-dependency catalog: `AppManifest`, compile-time registration/discovery, and a pure `resolveAppRoute(appId, isInstalled)` that takes install status as an input rather than looking it up itself. Empty of real registrations until Phase 3.
- **`src/connectors`** — pure adapters only (`ConnectorContract`: connect/disconnect/sync/onWebhook — receive input, validate, normalize, return output). No Firestore, no Core, no Platform import; the one stub (`custom-api`) proves the contract with zero real external system. Only `platform/connector-connections` is permitted to import this module.
- **`src/settings`** — Server Actions, forms, and pages only, calling into Platform's services. No business logic lives here.
